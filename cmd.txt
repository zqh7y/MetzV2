SOCKET = соединение, которое не закрывается, поэтому сервер сам шлёт обновления браузеру без перезагрузки.
Файл socket_server.py — отдельная программа (python socket_server.py), порт 8765 = websocket для браузеров, порт 8766 = Flask стучится сюда, чтобы запустить рассылку.
Поток: жмёшь "Join" → Flask обновляет базу → POST на :8766/broadcast → socket_server рассылает всем подписанным на эту встречу → счётчик обновляется сам у всех.
Демо: python app.py + python socket_server.py, открыть 2 разных браузера, в одном Join — во втором счётчик обновится сам.

ИФРОВАНИЕ ДАННЫХ (AES, не RSA)
------------------------------------------------
Использован AES (через Fernet из библиотеки cryptography) — это симметричный шифр, один и тот же ключ шифрует и расшифровывает.
RSA не подходит: он асимметричный (пара публичный/приватный ключ), нужен для обмена ключами между двумя сторонами или подписи, а не для шифрования файла на диске.
Ключ AES лежит в .env как DATA_ENCRYPTION_KEY (туда же, где остальные секреты), в код не зашит.
data.py: save_data() теперь шифрует JSON перед записью в app_data.json, load_data() расшифровывает при чтении.
Старый незашифрованный app_data.json автоматически прочитан один раз и пересохранён уже в зашифрованном виде.



app data → теперь хранится в настоящей MySQL базе (metz_app), user credentials/auth → Firebase.
(До этого было: encrypted app_data.json, потом SQLite — теперь полноценный MySQL-сервер, см. секцию "SQL (MySQL) В ПРОЕКТЕ" ниже.)



TCP В ПРОЕКТЕ
------------------------------------------------
TCP есть в socket_server.py: websockets.serve(...) держит открытый TCP-порт 8765 для браузеров (websockets.serve, строка ~120) и HTTPServer держит TCP-порт 8766 (строка ~109), куда Flask стучится через POST для запуска рассылки.
TCP (Transmission Control Protocol) — это протокол, который держит соединение открытым и гарантирует доставку данных в правильном порядке; именно поэтому через него можно слать обновления в любой момент без перезагрузки страницы.
Сами вызовы socket.socket() не используются — TCP-сокет создают библиотеки websockets и http.server под капотом.



ПРИМЕРЫ — КАК ЭТО РАБОТАЕТ В РЕАЛЬНОМ КОДЕ
------------------------------------------------

1) WebSocket / TCP, реальный поток (пример: юзер жмёт Join):
   - templates/base.html, строка 62: var ws = new WebSocket('ws://...:8765')
     — браузер открывает TCP-соединение на сокет-сервер при загрузке страницы.
   - routes/joined.py, join_route() (строка 69) вызывается когда юзер жмёт "Join".
     Он обновляет базу (toggle_join_meeting), потом строка 77:
     _notify_socket_server(meeting_id, {"count": ..., "joined_preview": ...})
     — это POST на http://127.0.0.1:8766/broadcast (см. _notify_socket_server, строка 9).
   - socket_server.py, BroadcastHandler.do_POST() (строка 80) принимает этот POST
     и вызывает broadcast(meeting_id, data) (строка 61), который шлёт
     ws.send(message) всем подписчикам этой встречи (строка 71).
   - templates/base.html, ws.onmessage (строка 70) получает сообщение и обновляет
     счётчик на странице без перезагрузки — document.querySelector(...).textContent = data.count.

2) AES-шифрование, реальный поток (пример: кто-то создаёт встречу):
   - data.py, save_data() сериализует MEETINGS_DB и USERS_DB в JSON,
     потом _fernet.encrypt(payload) шифрует это AES-ключом из .env
     (DATA_ENCRYPTION_KEY) и пишет байты в app_data.json.
   - При следующем запуске app.py, load_data() читает файл и делает
     _fernet.decrypt(raw) — если ключ верный, получает обратно тот же JSON.
   - Пример: открыть app_data.json в блокноте — видно нечитаемую строку
     вида "gAAAAABq..." вместо обычного {"meetings": {...}}.

3) Firebase Auth, реальный поток (пример: юзер логинится):
   - routes/login.py отправляет email+password на
     identitytoolkit.googleapis.com/.../signInWithPassword (HTTPS, тоже TCP).
   - Firebase сам сравнивает пароль с хэшем, который он хранит у себя,
     и возвращает токен — приложение никогда не видит/не хранит хэш пароля.

4) requirements.txt — websockets==13.1: эта строка означает, что библиотека
   websockets (она и создаёт TCP-сокет в socket_server.py) ставится через
   pip install -r requirements.txt, без неё websockets.serve(...) не запустится.



SQL (MySQL) В ПРОЕКТЕ
------------------------------------------------
Раньше все данные (meetings/users) лежали в файле (app_data.json, потом
app_data.db/SQLite). Теперь это настоящий MySQL-сервер (mysqld), не встроенный
файл — нужен реально запущенный сервер MySQL, как требовалось.
Установка на этом компьютере: MySQL Server 8.4 поставлен через winget
(Oracle.MySQL), данные лежат в meetupApp/mysql_data (не в Program Files —
туда нет прав на запись без админа). Сервер запускается командой:
  "C:\Program Files\MySQL\MySQL Server 8.4\bin\mysqld.exe" --datadir="<путь>\meetupApp\mysql_data" --port=3306
База данных называется metz_app (создана через CREATE DATABASE metz_app).
Драйвер для Python — pymysql (requirements.txt, чистый Python, без admin-прав
для установки). Подключение через переменные в .env: MYSQL_HOST, MYSQL_PORT,
MYSQL_USER, MYSQL_PASSWORD, MYSQL_DATABASE.
Таблицы (data.py, _get_connection()): meetings(id, data), users(uid, data),
meta(key, value) — реальные SQL-команды: CREATE TABLE IF NOT EXISTS,
INSERT INTO ... VALUES (%s, %s), SELECT id, data FROM meetings, DELETE FROM ...,
ON DUPLICATE KEY UPDATE (это MySQL-аналог INSERT OR REPLACE из SQLite).
Поле data в каждой строке — это JSON встречи/юзера, зашифрованный тем же
AES-ключом (Fernet) перед записью в базу — SQL и шифрование работают вместе:
MySQL хранит строки, AES не даёт прочитать их содержимое в открытом виде.
Старые app_data.json/app_data.db читаются один раз при первом запуске после
обновления (_migrate_legacy_data в data.py) и переносятся в MySQL автоматически.
MEETINGS_DB и USERS_DB остаются обычными Python-словарями в памяти — весь
остальной код (routes/joined.py и т.д.) их не заметил, изменился только
способ сохранения на диск внутри save_data()/load_data().
ВАЖНО для школьного компьютера: MySQL-сервер нужно отдельно ставить и запускать
там же (или поднять облачный MySQL и просто поменять MYSQL_HOST в .env) —
сам факт того, что там были заблокированы права админа (PowerShell execution
policy, поддельный python.exe), может помешать локальной установке MySQL.
