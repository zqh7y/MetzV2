## Часть 2: Сети, TCP/UDP, WebSocket

**8. Использует ли приложение протокол TCP или UDP?**
tcp медленный но точный udp быстрый но не точный!!!

TCP: гарантирует доставку, порядок пакетов и проверку ошибок — надёжно, но немного медленнее из-за этой проверки.
UDP: очень быстрый и лёгкий (без подтверждений), отлично подходит для видео/игр, где скорость важнее, чем потеря пары кадров.

Итог: выбрали TCP, потому что в приложении важна точность данных (счётчик участников не может "потеряться" или прийти в неправильном порядке) — а не максимальная скорость, поэтому надёжность TCP важнее, чем выигрыш в скорости от UDP.

**9. Почему не UDP для live-обновлений?**
UDp не точный, а быстрый не подходит для обновления информации.

**10. Что такое socket_server.py и почему он существует отдельно от Flask?**
socket_server.py это Отдельный TCP-сервер.
socket_server.py это websocket который работает в ручную без библиотеки pip install websockets.
для трансляции live-обновлений "количества участников" без блокировки основного Flask-сервера.

отдельно от flask:
Отдельный процесс, чтобы постоянные WebSocket-соединения не блокировали и не мешали обычным HTTP-запросам Flask.

и TAK браузеры не могут сами отправлять broadcast.

**13. Как происходит WebSocket handshake?**
По RFC 6455: клиент отправляет HTTP upgrade-запрос с заголовком Sec-WebSocket-Key, сервер вычисляет SHA1 от (key + фиксированный GUID) и возвращает Sec-WebSocket-Accept с кодом ответа 101 (200).

> 📄 `socket_server.py:57-64`:
> ```python
> accept = base64.b64encode(
>     hashlib.sha1((key + WS_HANDSHAKE_GUID).encode()).digest()
> ).decode()
> response = (
>     "HTTP/1.1 101 Switching Protocols\r\n"
>     f"Sec-WebSocket-Accept: {accept}\r\n\r\n"
> )
> ```

**14. Что такое thread?**
Thread (поток) — это "параллельная линия выполнения" позволяющая делать несколько вещей одновременно, не дожидаясь, пока закончится предыдущая задача.

socket_server.py: threading.Thread(target=handle_client, args=(conn,)).start()

> 📄 `socket_server.py:179` — `threading.Thread(target=handle_client, args=(conn, addr), daemon=True).start()`

**14. Что такое frame?**
Когда сервер хочет отправить обновление счётчика участников браузеру, он не может просто отправить текст "5" — он должен обернуть его в правильный frame (с заголовком 0x81 = "это текстовое сообщение"), иначе браузер не поймёт, что пришло.
server request "5" => "0x81"

> 📄 `socket_server.py:107-117` — функция `_encode_ws_frame`, строка 112: `header = bytes([0x81, length])`

**15. В чём разница между маскированным (masked) и немаскированным фреймом?**
Клиенты (браузеры) отправляют маскированные фреймы (XOR с 4-байтовым ключом) для предотвращения cache poisoning (атака хакеров);
серверы отправляют немаскированные - это разрешено спецификацией.

**16. Как кодируется фрейм WebSocket (encode)?**
Первый байт 0x81 (FIN+TEXT opcode), затем длина payload (1, 2 или 8 байт в зависимости от размера), затем сам контент в UTF-8.

обьяснение:
Возьмём реальное сообщение из твоего проекта: {"meeting_id": 5, "count": 4} — это 29 символов (29 байт в UTF-8).
Что отправляется по сети (в байтах):

0x81 0x1D { " m e e t i n g \_ i d " : 5 , ...
↑ ↑ └────────────── сам текст JSON ───────────────┘
заголовок длина
(FIN+TEXT) (29 в hex = 0x1D)

**17. Почему есть 3 разных формата для длины фрейма (≤125, ≤65535, больше)?**
Экономия пропускной способности - маленькие сообщения (как обновление счётчика) не нуждаются в 8 байтах накладных расходов на длину.
В некоторых случиях нету нуждности в 8 байтах в длинну иногда хватает и 3 байта длинны.

**18. Как работает механизм подписки на комнаты (rooms)?**
Глобальный словарь rooms сопоставляет meeting_id с набором подключённых sockets; браузер отправляет {"action":"subscribe","meeting_id":X} при входе на страницу и регистрируется в соответствующей комнате.

> 📄 `socket_server.py:31-32, 120-122`:
> ```python
> rooms = {}  # meeting_id -> set of raw client sockets subscribed to that meeting
> def register(conn, meeting_id):
>     with rooms_lock:
>         rooms.setdefault(meeting_id, set()).add(conn)
> ```

**19. Почему нужен threading.Lock для словаря rooms?**
Потому что несколько threads (один на каждого клиента) одновременно читают и пишут в общий словарь - без блокировки может возникнуть race condition или повреждение данных.

Представь, что 3 пользователя одновременно открыли страницу мероприятия №5 — для каждого создаётся свой thread, и каждый thread одновременно пытается добавить себя в rooms[5]:
Без Lock, если два thread'а делают это в одну и ту же миллисекунду, может случиться так, что один thread перезапишет изменения другого — и в итоге в rooms[5] останется только один пользователь вместо двух, хотя оба подписались.

> 📄 `socket_server.py:33` — `rooms_lock = threading.Lock()`

**20. Как происходит broadcast всем подписчикам комнаты?**

1- Пользователь А нажимает "Join" на мероприятии №5.
2- Сервер смотрит в rooms[5] — список всех браузеров, у кого сейчас открыта страница с этим мероприятием (например, пользователи Б, В, Г).
3- Сервер один раз готовит сообщение {"meeting_id": 5, "count": 4} (как "листочек с новостью").
4- Сервер проходит по списку (Б, В, Г) и каждому отправляет этот же листочек:

> 📄 `socket_server.py:131-143` — функция `broadcast(meeting_id, payload)`:
> ```python
> with rooms_lock:
>     subscribers = list(rooms.get(meeting_id, ()))
> frame = _encode_ws_frame(json.dumps(payload))
> for conn in subscribers:
>     conn.sendall(frame)
> ```

**21. Что происходит, если socket "мёртв" (disconnected) во время broadcast?**
1- Python выдаёт ошибку (OSError)
2- Код это "ловит" и не падает, а просто удаляет этого мёртвого пользователя из списка подписчиков (rooms)

Перехватывается OSError и вызывается unregister(conn), которая удаляет его из комнаты - автоматически очищает мёртвые подключения.

> 📄 `socket_server.py:140-143`:
> ```python
> try:
>     conn.sendall(frame)
> except OSError:
>     unregister(conn)
> ```

Чтобы если socket-сервер не работает, операция присоединения к встрече всё равно завершилась успешно (best-effort), а не зависла в ожидании ответа.

> 📄 `routes/joined.py:13-16`:

**24. Каждый клиент получает отдельный thread - почему не asyncio?**
простота - модель threading легче понять и реализовать в учебном проекте - даже если asyncio более масштабируем.

**26. В чём разница между Flask (HTTP) и socket_server (чистый WebSocket) в плане коммуникации?**
HTTP это однонаправленный (|/|) request-response, который закрывается в конце, а
WebSocket это открытое двунаправленное (|/| |\|) TCP-соединение, позволяющее серверу отправлять сообщения без запроса клиента (push).

**27. Почему нельзя обновлять счётчик участников только с помощью HTTP polling?**
Polling требует, чтобы клиент снова и снова отправлял новые запросы (накладные расходы и задержка), в то время как WebSocket позволяет немедленное и более эффективное обновление по сети.

**34. Почему /join и /pass - это POST, а не GET?**
GET не должен вызывать изменение состояния на сервере (idempotent), POST подходит для операций, изменяющих данные (стандарт REST).

> 📄 `app.py:85` и `app.py:90`:
> ```python
> @app.route("/pass/<int:meeting_id>", methods=["POST"])

**34. что такое POST и GET?**
GET и POST — это два типа HTTP-запросов, которые отличаются по назначению.

GET — запрос данных ("дай мне страницу/информацию"), не должен менять что-то на сервере.
POST — отправка данных на сервер для изменения чего-то (создать, изменить, удалить).

**49. Почему таблицы (meetings, users) хранят LONGBLOB, а не обычные столбцы (title, email и т.д.)?**
Данные зашифрованы (Fernet) перед сохранением, поэтому их нельзя хранить как обычный текст в отдельных столбцах

**48. что такое LONGBLOB?**
LONGBLOB — это тип столбца в MySQL для хранения больших "бинарных" данных (просто кучи байтов), а не обычного текста.

(data.py) он используется так:

**50. Какой недостаток подхода с зашифрованным blob с точки зрения SQL?**
Невозможно выполнять SQL-запросы (WHERE) по содержимому данных (например, поиск по имени) - вся фильтрация должна происходить в памяти после расшифровки.

- при большом количестве пользователей это медленнее, чем обычный SQL-поиск, потому что приходится перебирать всё вручную в памяти, а не доверять это базе данных.

> 📄 `data.py:71` — шифруется весь JSON целиком: `_fernet.encrypt(json.dumps(m).encode("utf-8"))` → нет отдельных столбцов для WHERE-поиска

**52. Почему есть отдельная таблица meta?**
Для хранения общих метаданных (например, next_meeting_id), не принадлежащих конкретной сущности (ни meeting, ни user).

meta хранит общие числа/настройки приложения (например, "какой ID выдать следующему мероприятию"), которые не относятся ни к конкретному пользователю, ни к конкретному мероприятию.

> 📄 `data.py:52-56`:
> ```python
> cur.execute("""
>     CREATE TABLE IF NOT EXISTS meta (
>         `key` VARCHAR(64) PRIMARY KEY,

## Часть 5: Безопасность и шифрование (Cryptography)

**67. Какой метод шифрования используется для хранения данных в БД?**
Fernet из библиотеки cryptography - симметричное шифрование на основе AES (128-бит CBC) с HMAC для проверки целостности.

**68. Почему выбран Fernet, а не "сырой" AES?**
Fernet - это безопасный high-level интерфейс (автоматически обрабатывает IV, padding и MAC), снижает риск ошибок реализации, ведущих к уязвимостям.

**69. Где хранится ключ шифрования (DATA_ENCRYPTION_KEY) и почему не в коде?**
В переменной окружения (.env), чтобы не попасть в git (исключён через .gitignore) - ключ, открытый в публичном коде = все данные раскрыты.

**70. В чём разница между симметричным шифрованием (Fernet) и асимметричным?**
Симметричное - один и тот же ключ для шифрования и расшифровки (быстрее, подходит для собственного хранения);
асимметричное - отдельные публичный/приватный ключи (подходит для коммуникации между сторонами, не имеющими общего секрета заранее).

Здесь подходит симметричное, потому что только сервер читает и пишет.

**71. Хранятся ли пароли пользователей в приложении?**
Нет - Firebase обрабатывает хеширование (bcrypt с salt) и хранение;
сервер никогда не видит пароль в открытом виде (plaintext).

**72. Что такое idToken и какова его роль?**
JWT-токен, возвращаемый Firebase после успешного login/signup, подтверждающий, что пользователь прошёл аутентификацию; сохраняется в session.

**72. Что такое JWT token?**
JWT (JSON Web Token) — это специальный токен, который подтверждает "этот пользователь прошёл аутентификацию",
без необходимости каждый раз заново проверять пароль.

В твоём проекте это idToken, который возвращает Firebase после логина:

**73. Почему выбран Firebase Identity Toolkit вместо собственной реализации аутентификации?**
Не изобретать механизм безопасности самостоятельно - Firebase обрабатывает хеширование,

**74. Какой алгоритм хеширования используется для создания UID из email?**
SHA256 - однонаправленный (необратимый) хеш, генерирующий детерминированное число из email для построения короткого уникального идентификатора.

> 📄 `data.py:424-429` (`generate_user_id`):
> ```python
> def generate_user_id(email):
>     email = email.strip().lower()
>     num = int(hashlib.sha256(email.encode()).hexdigest(), 16) % 10000
>     prefix = email.split("@")[0][:3].upper()
>     return f"{prefix}{num:04d}"
> ```

**75. Почему SHA256, а не MD5 для создания UID?**
SHA256 сильнее и современнее; MD5 считается скомпрометированным (broken) для коллизий и не рекомендуется для нового использования, даже для такого некриптографического случая.

> 📄 `data.py:427` — `hashlib.sha256(email.encode()).hexdigest()`

Чтобы пользователь, помеченный как banned, не мог зарегистрироваться снова с тем же email, но другим регистром букв (Art@x.com против art@x.com) - предотвращает "обход бана".

## Бонус: общие вопросы "дизайн/почему"

**101. Почему не выбрано использование ORM (например, SQLAlchemy)?**
Единственная модель - это "общий JSON-документ" (key-value), поэтому ORM, подходящий для структурированных столбцов, добавляет ненужные накладные расходы; простая таблица blob+key достаточна.

**102. Что бы вы изменили в архитектуре при продолжении разработки (масштабирование)?**
Переход на настоящую БД со структурированными и индексируемыми столбцами (для поиска), Redis Pub/Sub вместо ручного socket-сервера, и проверенная WebSocket-библиотека вместо собственной реализации RFC 6455.

**ПРОБЛЕМЫ ВО ВРЕМЯ РАЗРАБОТКИ**

**1. Проблема: обновление счётчика участников требовало постоянной перезагрузки страницы.**
Сначала я хотел просто перезагружать страницу при каждом действии (Join/Pass), но это было неудобно и медленно для пользователя. Решение: написал отдельный socket_server.py с WebSocket, чтобы сервер сам "толкал" обновления браузеру в реальном времени без перезагрузки.

**2. Проблема: данные пользователей хранились в открытом виде в базе.**
На раннем этапе данные (email, мероприятия) сохранялись как обычный JSON без защиты — если бы кто-то получил доступ к базе, увидел бы всё напрямую. Решение: добавил шифрование Fernet (AES) перед записью в MySQL, так что даже при утечке базы данные останутся нечитаемыми без ключа.

**4. Проблема: новые пользователи создавали мероприятия, которые сразу появлялись для всех, включая спам/неподобающий контент.**
Не было модерации — любой зарегистрированный мог сразу опубликовать что угодно. Решение: добавил систему статусов "pending/approved" — мероприятия от обычных пользователей сначала идут на проверку администратору, а доверенные (trusted) пользователи публикуют сразу.
